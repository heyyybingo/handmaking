## Context

当前代码库处于早期开发阶段，业务模块和基础组件均已搭建好模块骨架。现有代码中注释覆盖不足：部分 controller/service 缺少类和方法级别的说明注释，复杂逻辑缺少解释性注释，公共基础设施（filters、interceptors 等）的用途和使用方式未在代码中说明。此外，项目缺少统一的注释规约，后续多人协作时注释风格可能不一致。

注释规约需要同时覆盖 NestJS 后端和 React 前端，使用中文注释（与项目沟通语言保持一致）。

## Goals / Non-Goals

**Goals:**
- 为所有已存在的业务模块代码添加合适的注释（类注释、公共方法注释、复杂逻辑注释）
- 为公共基础设施代码添加注释说明用途和使用场景
- 制定注释规约并写入 CLAUDE.md，作为项目级别编码约束
- 注释使用中文，简洁明了

**Non-Goals:**
- 不添加 JSDoc/TSDoc 格式的 API 文档生成型注释（Swagger 已通过装饰器覆盖）
- 不为自动生成的代码（如 migration 文件）添加注释
- 不为测试文件添加注释（测试用例名称即是文档）
- 不修改现有代码逻辑，仅添加注释

## Decisions

### 注释语言：中文

**决策**：所有注释使用中文。
**理由**：项目团队使用中文沟通，CLAUDE.md 已明确"始终使用中文"。中文注释能降低团队理解成本。

### 注释层级：三级注释体系

**决策**：采用三级注释体系——
1. **类/模块注释**：说明该类的职责、主要功能（放在类声明上方）
2. **公共方法注释**：说明方法用途、参数含义、返回值（放在方法声明上方）
3. **关键逻辑注释**：解释复杂算法、非显而易见的业务规则、边界处理（放在相应代码块上方或行内）
**理由**：三层分级避免过度注释（私有方法、简单 getter/setter 不需要），同时确保关键信息不缺失。

### 注释格式：JSDoc 风格的中文注释

**决策**：后端使用 JSDoc 风格的块注释（`/** ... */`），前端组件使用 JSDoc 风格或单行注释（`//`），都使用中文内容。
**理由**：JSDoc 风格在主流 IDE 中有良好的 hover 提示支持，有助于开发体验。

### 规约存放位置：CLAUDE.md

**决策**：将注释规约作为独立章节添加到 CLAUDE.md 中。
**理由**：CLAUDE.md 是项目的编码约束入口，AI 辅助开发和人类开发者都会参考。

### 实施策略：按模块分批推进

**决策**：按 server 模块 → common 基础设施 → storage → entities → web-admin 的顺序分批添加注释。
**理由**：从核心业务逻辑向外围推进，确保最重要的代码优先获得注释覆盖。

## Risks / Trade-offs

- **[风险] 注释与代码不同步**：后续代码修改可能使注释过时 → 代码审查时检查注释同步性，CLAUDE.md 中明确注释维护要求
- **[权衡] 注释增加文件长度**：注释会增加代码行数 → 只注释必要的层级（类、公共方法、关键逻辑），避免过度注释
- **[权衡] 中文注释对国际化不友好**：如果项目未来需要开源或国际化 → 当前阶段团队以中文为主，此问题暂不考虑
